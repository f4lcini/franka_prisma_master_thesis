# Custom Bimanual Franka Gazebo Simulation Setup

This document serves as a comprehensive walkthrough of the work done to integrate and customize the bimanual Franka Research 3 simulation in Gazebo, tailored for execution within a Docker environment. You can use this guide to recall the implemented features, test commands, and troubleshooting fixes.

## ­¤ÜĆ Implemented Modifications

### 1. Custom Gazebo World (`bimanual_custom.world`)
To avoid modifying the native `idra_franka_launch` package, a custom world file was created inside `franka_manipulation_env/worlds/`.
*   **Base Setup**: Includes the standard Ignition/Gazebo plugins (Physics, Sensors, Scene Broadcaster), a ground plane, and directional sunlight.
*   **Custom 3D Objects**: 
    *   Added a dynamic **`green_ball`** (mass: 0.2kg, radius: 0.05m).
    *   Added a static **`open_box`**, built by intersecting 5 standard `<box>` geometries (a base and four walls).
    *   *Pose Adjustments*: By calculating the logical Z-offset generated by the URDF `table` link (`Z=0.2`), both objects were rigidly positioned exactly **on** the desktop surface so they don't fall through the floor.
*   **YOLO Camera Integraton**: Instantiated a static `<model name="camera_model">` enclosing an Ignition RGB sensor (`type="camera"`). It faces the table top-down (at `Y=-0.4, Z=0.8`, pitched downwards by 0.5 rad) to publish a simulated `640x480` visual frame at `30fps` directly onto the ROS 2 topic `/camera/image_raw`.
*   **Environment Scaling**: Expanded the `ground_plane` collision mesh to a massive 10x10 meters.

### 2. Dedicated Launch File (`custom_bimanual.launch.py`)
A custom Python launch file was drafted under `franka_manipulation_env/launch/` to inject the new context.
*   **Gazebo Resource Pathing**: Altered the `GZ_SIM_RESOURCE_PATH` / `IGN_GAZEBO_RESOURCE_PATH` environment variables at runtime to ensure the backend locates the custom `/models` directory implicitly installed by `franka_manipulation_env`.
*   **World Binding**: The launch string now explicitly mounts `bimanual_custom.world` instead of the generic `empty.sdf`.

## ­¤É× Bugs Fixed (Docker & ROS 2 Compatibility)

During this setup, a variety of crashes native to Docker networking and generic ROS 2 Humble logic were identified and resolved in the launch files:

1.  **Gazebo Docker Loop**: Forced `IGN_IP` and `GZ_IP` to `127.0.0.1` natively. This resolves the `Requesting list of world names` loop caused by bridge network mismatches inside containers.
2.  **Controller Timeouts**: Increased the TimerAction wait time from 2.0s to 10.0s. This grants the `ros_gz_sim create` node ample time to heavily parse the massive URDF tree before the `controller_manager` spamming lists controllers.
3.  **libfranka Connection Crash**: Hardcoded an `IfCondition` using `PythonExpression` to automatically disable the real `franka_gripper_node` whenever `use_gazebo:=true` is evaluated. This prevents the simulation from freezing while attempting to search the LAN network for physical gripper IPs.
4.  **SDF Missing URIs (`Unable to find uri[model://cube]`)**: The launch file explicitly defines the `model://` folder routing within the Gazebo environment script, and old missing models were pruned from the XML world file. 
5.  **Joint State Publisher Cast Error**: Bypassed a C++ parameter type-cast crash on `rclpy` arrays by injecting `source_list` string paths inline for the grippers (`franka1/franka_gripper/joint_states`, etc.).
6.  **Gazebo Plugin Conflict**: Removed an explicit `<plugin filename="ign_ros2_control-system" ...>` in the custom `.world` file as it violently conflicted with the exact same namespace already auto-generated by the URDF macro `gazebo_ros2_control`.
7.  **`basic_controllers.yaml` Parsing Error**: Restored the conventional YAML indentation structure since `rclcpp` Controller Manager was crashing fatally while loading `pose_reader` and `franka1_arm_controller` properties.
8.  **"Not existing" Hardware Interfaces**: Stripped default attempts to load Cartesian impedance controllers in simulation. Since `ros_gz_sim` plugins do not export advanced cartesian control (libfranka calculates it on physical arms), the launch file now correctly spawns **joint space controllers** (e.g., `franka1_arm_controller`, aka `joint_trajectory_controller`) if `use_gazebo:=true`.

## ŌÜĀ’ĖÅ Known Recurring Docker Issue

**Infinite Loop `"Requesting list of world names"` after a crash:**
*   **Why it happens:** If Gazebo goes into a fatal state once within your Docker container, the ROS parent dies, but a zombie child process (`ign` or `ruby`) remains pinned to the local network port natively. The next time you fire up the launch script, it will forever try to contact this "ghost" server and simply hang.
*   **The Solution:** Always execute `killall -9 ruby ign gazebo` on the terminal or effortlessly restart the whole Docker Container (`docker restart ...`) to reset the port bindings when testing changes.
